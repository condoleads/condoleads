The function body is PURELY PARAMETER-DRIVEN:
  - Reads its 4 input parameters (p_listing_id, p_tenant_id, p_is_condo, p_is_home)
  - SELECTs from tenant_floor_pool + agents (filtered by p_tenant_id, is_active, is_selling)

  - Returns the hash-RR picked agent_id

No path reads the calling role's identity, JWT, or session state. Switching
from INVOKER to DEFINER changes only WHICH role's grants/RLS apply during the
function's own queries — it does not change the function's outputs given the
same inputs.

--- Callers' security model (both INVOKER) ---

reroll_listings_at_floor   SECURITY INVOKER
resolve_agent_for_context  SECURITY INVOKER

Postgres semantics for INVOKER -> DEFINER call:
  1. Caller's auth/role context is preserved up to the call.
  2. Inside pick_floor_agent body, role temporarily becomes the function's
     OWNER (postgres) for the duration of the body.
  3. Body returns; control returns to caller; caller's role context resumes
     unchanged.
  4. Subsequent SQL in the caller still runs as the original invoker.

No "chain break" — INVOKER->DEFINER->INVOKER chaining is the standard
Supabase pattern (helper functions with SECURITY DEFINER for trusted DB
operations called from RLS-bound INVOKER functions). resolve_agent_for_context
continues to operate as INVOKER, sees the caller's auth.uid() correctly,
enforces tenant scoping via its p_tenant_id parameter. Only the floor-branch
SELECT inside pick_floor_agent's body runs with elevated privs.

VERDICT — PRECONDITION 2: PASS.
SECURITY DEFINER on pick_floor_agent is surgical. The two callers don't rely
on the failing privilege path; they delegate the floor-resolution entirely.
