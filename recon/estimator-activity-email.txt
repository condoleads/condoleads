HEADLINE — the hypothesis is FALSE.
═══════════════════════════════════════════════════════════════════════════

  lib/email/sendActivityEmail.ts has ZERO production callers. It's
  DEAD CODE — only the function definition (L320) exists; nothing in
  app/, components/, or lib/ imports or calls it. The header comment
  at lib/actions/leads.ts:8 records that it was REPLACED:
    "Email fan-out replaced: was 3 sequential sendActivityEmail calls
     (agent → manager → admin loop with receive_*_emails flags),
     now single getLeadEmailRecipients + sendTenantEmail with TO/CC/
     BCC (open chain, no suppression per Q1)"

  The Offer/Estimate buttons on the property page do NOT fire
  sendActivityEmail. They fire TWO server actions:
    1. submitLeadFromForm (creates a leads row + email fan-out via
       getLeadEmailRecipients in lib/actions/leads.ts — the
       contract-conformant path verified in the prior recon).
    2. submitActivityFromForm (writes a user_activities row only;
       no email cascade — trackActivity:128-129 says explicitly
       "Actual email cascade lives in getOrCreateLead, not here").

  BOTH server actions read headers().get('x-tenant-id') and return
  {success:false} on null — same gate as the estimator's
  submitLeadFromForm. Pre-e79c670 the gate fired, BOTH actions
  refused the work, NEITHER lead NOR activity got written. That
  explains the live DB pattern:
    Last sale_offer_inquiry activity under WALLiam:  2025-12-21
    Last lease_offer_inquiry:                          2025-12-16
    Last estimator_used:                               2025-11-26
    Last estimator_contact_submitted:                  2026-04-17
    Activities in scope, last 24h:                     ZERO

  The offer + estimate path has been dead for months because BOTH
  server actions silently early-exit on missing x-tenant-id, same
  root cause e79c670 addresses.

  Additional finding: sendActivityEmail (the dead code) is also
  fundamentally SYSTEM 1 — it uses process.env.RESEND_API_KEY +
  HARDCODED `from: 'CondoLeads <notifications@condoleads.ca>'` +
  HARDCODED `https://condoleads.ca/dashboard/...` URL. Even if
  it had callers, it would send from condoleads.ca for any tenant,
  including walliam. It is not a viable hierarchy-fan-out path.

═══════════════════════════════════════════════════════════════════════════
ROOT CAUSE
═══════════════════════════════════════════════════════════════════════════

  The offer/estimate activity-email path "fails" at TWO server-
  action null-checks that fire when x-tenant-id is absent from the
  request header:

    app/actions/submitLeadFromForm.ts:50-58
        const tenantId = headersList.get('x-tenant-id')
        if (!tenantId) {
          return { success: false, error: 'Tenant context unavailable...' }
        }

    app/actions/submitActivityFromForm.ts:32-40
        const tenantId = headersList.get('x-tenant-id')
        if (!tenantId) {
          return { success: false, error: 'Tenant context unavailable.' }
        }

  Both fire on EVERY button submit on walliam.ca page routes until
  middleware.ts started injecting x-tenant-id on server-action POSTs
  (the e79c670 commit). The cliff dates in R3 (Dec 2025 – April 2026)
  align with the W-TENANT-AUTH Phase 3 migration that moved these
  callers from /api/ routes to server actions — the new server-
  action path was never paired with a middleware update for that
  pathname class, so they have been silently failing since.

  The operator's "estimator-via-activity-email" hypothesis is wrong:
    • sendActivityEmail has no production callers (dead code).
    • The activity row insert is gated by the same x-tenant-id check
      as the lead insert.
    • The email cascade for offer/estimate has ALWAYS been routed
      through lib/actions/leads.ts:createLead (the contract-conformant
      helper path), not through sendActivityEmail.

  The fix e79c670 addresses the SHARED root cause for:
    • lead row creation (submitLeadFromForm)
    • activity row creation (submitActivityFromForm)
    • email cascade (downstream of lead, via the helper)
  All three legs unblock at once when x-tenant-id reaches the
  server action.
