CV source-grep verdict (operator's #3 question)
───────────────────────────────────────────────────────────────────────────
CV-3 was source-static — it asserted that the JSX/HTML strings exist
in the surface source code, and that data fields reference the canonical
view shape. It did NOT exercise the runtime data path. CV-3 PASSED on
all 14 sections × 3 surfaces because:
  - lead page, email, in-chat ALL have the section markup in source
  - the section markup reads the right view.* fields
  - tier-color literals are sourced from CV-0 TIER_META

That is necessary but NOT sufficient. CV-3 cannot detect:
  - matcher returning empty taxMatch.comparables → silent gate (the
    operator's likely scenario)
  - runtime conditional rendering hiding a section (e.g.
    ResultsPanel:107 buyer-vs-seller gate)
  - data-shape mismatch between matcher output and view fields
  - cached/stale browser state showing an old render

REAL-CHARLIE (this harness) is the runtime ground truth: it observes
what the chromium DOM actually contains after the seller flow completes.

WHAT CV-3 GOT RIGHT, WHAT IT GOT WRONG
═══════════════════════════════════════════════════════════════════════════

GOT RIGHT (renderer source claims, confirmed by live DOM):
  - lead-page / email / in-chat all have the section markup
  - tier colors sourced from CV-0 TIER_META (no inline drift)
  - in-chat DELIBERATE-GATE classifications for market_intel /
    price_by_home_type / offer_intel — confirmed gated live

GOT WRONG (or insufficient):
  - "PRESENT" in source DID NOT MEAN "PRESENT in rendered DOM" — a
    conditional gate, a missing data field, or a different label can
    silently suppress a source-grep-positive section. CV-3 cannot see
    this.
  - "seller_profile = PRESENT in in-chat" was a source-string match for
    something the in-chat surface labels DIFFERENTLY at runtime ("Your
    Property" / "Market Snapshot"). The CV-3 harness checked the wrong
    surface-specific label.

WHAT THIS CHANGES FOR THE PROGRAM
───────────────────────────────────────────────────────────────────────────
Source-static convergence (CV-3) is a necessary backstop — it catches
markup drift, helper rename, tier-color regression. But it CANNOT be
the only proof. A real-render harness like this one MUST run against
every deploy before claiming "no regression in the seller panel". CV-3
is the unit-test layer; REAL-CHARLIE is the integration-test layer.

Recommendation (out of scope to implement):
  Add REAL-CHARLIE-1 (in-chat, this harness) +
      REAL-CHARLIE-2 (lead page) +
      REAL-CHARLIE-3 (email-client render)
  as a 3-surface real-render triple, gated as a release-blocker.
  CV-3 stays as a fast-feedback CI test but is no longer the source
  of truth.
