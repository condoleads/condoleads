RECON 3 — FIX SHAPE (flag only, do NOT implement)
═══════════════════════════════════════════════════════════════════════════

  Q: Is plan_data available client-side at in-chat render time?
  A: NO. The persisted lead.plan_data is written by the SERVER inside
     plan-email/route.ts. The client has:
       • state.plan       = generate_plan tool result (thin metadata,
                            no listings/comparables/buyerTaxMatch)
       • state.listingGroups, state.comparables = tool-driven only
     There is currently no GET endpoint the client uses to read its
     own lead's plan_data back. A fetch is needed, OR the existing
     plan-email response needs widening.

  CANDIDATE FIX (cleanest, minimal, additive):
    Mirror the backfill outward. The server route already computes
    `effectiveListings` (the backfilled For-Sale set) and persists
    `plan_data.buyerTaxMatch` derived from those listings. Return
    both in the plan-email JSON response. Client dispatches them
    through the SAME state-update path search_listings + get_comparables
    use, so the existing in-chat render code path Just Works.

    THREE-LOC CHANGE FOOTPRINT (flag only):

      app/api/charlie/plan-email/route.ts:423-429
        Widen the existing return shape:
          return NextResponse.json({
            success: true,
            userEmailSent, userEmailReason,
            chainEmailSent, chainEmailReason,
            // NEW: pass the backfill's product to the client so in-chat
            // can converge with email + lead on a single source of truth.
            backfilledListings: effectiveListings ?? [],
            backfilledTaxMatch: buyerTaxMatch ?? null,
            // (and optionally backfilledComparables if get_comparables
            // also missed in-session — but those are NOT currently re-
            // computed in plan-email; would need a parallel comp fetch.)
          })

      app/charlie/hooks/useCharlie.ts:560-566 (the .then handler)
        On planEmailData with backfilledListings.length > 0 AND state.
        listingGroups empty, dispatch through the EXISTING search_listings
        path:
          setState(s => ({ ...s,
            listingGroups: [...s.listingGroups, { label: 'Matched Listings', listings: backfilledListings }],
            blocks: [...s.blocks, { type: 'listings', label: 'Matched Listings', listings: backfilledListings }],
          }))
        This makes the For-Sale block appear in-chat with the SAME
        listings email + lead already render — single source of truth.

      Tax-Match (the harder half):
        The Tax-Match in-chat block requires the 'comparables' block to
        exist (because it's rendered as its sibling). Two options:

        OPTION T-A — simpler: decouple BuyerTaxMatchInChat from the
          comparables-block branch in ResultsPanel.tsx:466-470. Render
          it as a TOP-LEVEL conditional block: whenever the buyer's
          plan is ready AND a tax-match result is available, render it
          standalone. Doesn't depend on get_comparables having fired.

        OPTION T-B — heavier: also backfill get_comparables on the
          server (parallel to listings backfill) and return
          backfilledComparables in the plan-email response. Then
          dispatch through the get_comparables push path
          (useCharlie.ts:610-637) so the 'comparables' block appears
          in-chat. Tax-Match block then renders inside it as before,
          via the existing self-fetch path.

        Recommendation: T-A. It also makes the in-chat Tax-Match block
        survive when ONLY comparables backfill is missing — strictly
        more robust. Email + lead don't care about this gate; they
        read plan_data.buyerTaxMatch directly.

  NO-REGRESSION SHAPE:
    • Server response shape becomes a SUPERSET (additive). Existing
      readers (the current useCharlie .then handler reads only
      `userEmailSent`) unaffected.
    • Client backfill hydration only fires when listingGroups is
      already empty — the WORKING in-session path (search_listings did
      fire) gets no behavior change. Single defensive flag.
    • Seller path: gate before the dispatch on plan.type === 'buyer'
      so seller plans still receive no listings hydration.
    • Persisted surfaces (email + admin lead) are READ-ONLY of the
      same plan_data they read today. Zero change in their inputs.
