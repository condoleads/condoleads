  2.1 — Condo location inputs in SellerForm TODAY

    SellerForm condo flow uses the IDENTICAL location inputs as the
    home flow (no condo-specific branch). At SellerForm.tsx:316-328:

      streetNumber  inp("No.")                — free-text input
      streetName    inp("Street Name")        — free-text input
      city          CitySearch (line 323-327) — typeahead against
                                                /api/charlie/municipalities
                                                returns [{id, name, displayName}]
                                                writes municipalityId on select

    The user is expected to KNOW their building's street address.
    There is NO building-name search and NO building autocomplete
    on SellerForm for condos. The address is resolved server-side
    in the seller-estimate route at L13-37: street_number + first
    word of street_name → `buildings.canonical_address ILIKE` fuzzy
    match; falls back to street_name + city_district ILIKE; returns
    first match or "No building found" error.

    Result: a user who knows their CONDO BUILDING NAME but not the
    canonical street address (or types it imprecisely) gets a
    "No building found" hard-fail from the API. There is no
    forgiving search-by-name path on the seller form.

  2.2 — EXISTING building-search capability (reusable, already shipped)

    YES — Charlie already has working building search wired into the
    BuyerForm. The component + endpoint are production-grade and can
    be reused as-is.

    Component: AreaSearch in app/charlie/components/BuyerForm.tsx:107-181
      Typeahead input that calls /api/search?q=… and renders grouped
      results: "Areas & Communities" + "Buildings" + "Active Listings"
      etc. Each result carries { type, id, name, slug, subtitle, url }.

                               listing:'🏠' } (BuyerForm.tsx:131-137)

    Endpoint: app/api/search/route.ts
      GET /api/search?q={query}
      Returns { groups: [{ label, results: SearchResult[] }] }.
      SearchResult shape (app/api/search/route.ts:18-25):
        { type, id, name, slug, subtitle, url }

      Search intents (route.ts:36-50 detectIntent):
        - number_only ("101"):  street-number → buildings_with_listing_counts
        - address ("101 King"): number+street → buildings + units

      ADDRESS mode (route.ts:158-282, condensed) is the by-street-number
      search:
        buildings_with_listing_counts
          .ilike('street_name', streetPattern)
          AND street_number matched separately
        Falls back to street-name-only when number+street finds nothing.

    Verdict: NAME-search AND STREET-search are both already implemented
    against the SAME `buildings_with_listing_counts` view, surfaced
    via a SINGLE endpoint, consumed by an ALREADY-SHIPPED autocomplete
    component (AreaSearch) that the BuyerForm uses. Reuse path is

  3.1 — Tenant scoping of /api/search

    File: app/api/search/route.ts
    Tenant filter on the buildings/communities/listings queries: NONE.

    Reading the entire file: the route uses a service-role Supabase
    client and queries buildings_with_listing_counts, communities,
    municipalities, neighbourhoods, mls_listings WITHOUT a tenant_id
    filter. This is INTENTIONAL for this endpoint — `/api/search` is
    the GLOBAL public search used by tenants' branded sites (the
    landing autocomplete on walliam.ca, aily.ca, etc.). All tenants
    see the same building catalog because the buildings table itself
    is tenant-agnostic (the GTA building registry).

    Middleware behavior (middleware.ts:108-115):
      For all /api routes the middleware injects `x-tenant-id` header
      based on the request host. /api/search does NOT consume this
      header today — confirmed by grep ("tenant" / "x-tenant" in
      route.ts → 0 matches).

    Implication for the fix:
      Reusing /api/search for SellerForm building search is
      TENANT-CORRECT by construction — the user types a building
      name on walliam.ca's Charlie panel, the global registry returns
      matches, and the user's choice writes building_id into form
      state. The subsequent seller-estimate API call IS tenant-aware
      (via x-tenant-id header injection), so the matcher's tenant
      scoping happens DOWNSTREAM of the search, exactly as it does
      for the home flow today.

    There is no current need to add tenant_id filtering to /api/search
    for this fix — buildings are not tenant-scoped data.

  3.2 — Geo scoping options for the building search

    If the operator wants the building search SCOPED by geo (e.g.
    "only show buildings in Pickering"), the search would need to
    either:
      (a) Accept a geo filter param (community_id or municipality_id)
          and filter the buildings_with_listing_counts query by it
          (existing column on the buildings table: community_id).
          Endpoint change: add `?geo_id=…&geo_type=community` query
          string + WHERE clause.
      (b) Stay global (status quo) and rely on the user to type
          enough of the building name to disambiguate.

    Status quo is what BuyerForm uses (global search; user-typed
    disambiguation). Wiring (a) into a new SellerForm condo search
    would need:
      - the seller form's CitySearch (which already returns
        municipalityId) to act as an UPSTREAM geo filter on the
        building autocomplete
      - /api/search to accept the geo filter param

    The BuyerForm's geo answer can be a BUILDING directly — the
    AreaSearch is the FIRST and ONLY location input. Seller's
    different ordering (city → street → building?) is a design
    decision; both shapes are supportable from the existing
    endpoint with minor or no changes.

  3.3 — Form-level location-mandatory enforcement

    SellerForm: street_number AND street_name AND city AND
    municipalityId are ALL required (line 269 — canProceed gate).
    The form will not submit until all four are filled, AND the city
    must be selected from the dropdown (CitySearch onSelect writes
    municipalityId; typing a city without selecting leaves
    municipalityId empty → canProceed false → button disabled).

    BuyerForm: form.area AND form.geoId required (line 231 —
    canSubmit). Same pattern: typing without selecting leaves
    geoId empty → button disabled. BuyerForm.tsx:274 surfaces an
    inline amber warning when this state is detected:
      {form.area && !form.geoId &&
        <div>⚠ Please select a location from the dropdown</div>}
    This is the only inline-validation hint in the codebase; could
    be cloned to seller for the same "typed but not selected" case.

    For a CONDO building-as-location pattern, the equivalent gate
    would be: form.buildingId required (analogous to geoId). The
    canSubmit logic would need to know whether the user picked a
    building vs typed a street.
