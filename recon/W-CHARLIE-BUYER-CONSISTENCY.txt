2.1 What deriveBuyerTaxMatch CURRENTLY computes — assessment framing

  lib/charlie/buyer-tax-match.ts:69-130

  Logic:
    1. Filter listings to those with tax_annual_amount > 0
    2. If fewer than MIN_WITH_TAX (3) → isEmpty=true
    3. medianTax  = quantile(sortedTax, 0.5)
       taxBand   = { low: quantile(.25), high: quantile(.75) }
       samples   = listings sorted by ABS(tax - medianTax), take 5,
                   sort that 5 by tax ascending

  Input set: MATCHED LISTINGS, which are ACTIVE for-sale listings
  (search_listings returns standard_status='Active' /
   transaction_type='For Sale').

  Output framing: "median annual tax + 25/75 band of what these ACTIVE
  listings carry as annual property-tax" — an ASSESSMENT range, not a
  set of SOLD-comp evidence.

2.3 CORRECT framing — recently SOLD comps matched by TAX BAND

  The seller-side matcher (lib/estimator/home-comparable-matcher-
   sales.ts) already implements exactly this for sellers:

    Function:  home-comparable-matcher-sales.ts:1220-1290 (the tax-
               cascade getter inside `getMatchedSales` or similar entry
               point — function signature shown at L1220 returning
               HomeMatchResult['taxMatch'])
    Inputs:    specs.subjectTaxAnnualAmount (the SELLER's subject tax)
    Constants: TAX_BAND_PCT       (e.g. ±20%; cited at L1242-1243:
               `taxLow = subjTax * (1 - TAX_BAND_PCT)`,
               `taxHigh = subjTax * (1 + TAX_BAND_PCT)`)
    Query:     mls_listings WHERE
                 community_id = ... AND
                 transaction_type = 'For Sale' AND
                 standard_status = 'Closed' AND        ← SOLD
                 close_price NOT NULL AND
                 close_date >= twoYearsAgo AND
                 tax_annual_amount BETWEEN taxLow AND taxHigh AND
                 tax_year BETWEEN yearLo AND yearHi
               ORDER BY close_date DESC
               LIMIT 300 (community pool) + 500 (muni pool)
    Output shape (interface HomeMatchResult['taxMatch'] at L120-132):
      { matchTier, comparables: ComparableSale[], winnerComparables,
        count, tiers, bestGeoTier }
      → comparables is a list of SOLD listings, each with full
        close_price + close_date + sourceTier stamps.

  Reusability for BUYER: YES. The function takes:
    - subjTax → which CAN be derived from buyer's matched-listings
                 median (or from the get_comparables result's own median).
    - tax_year window → derivable from "current year ± 1" since the
                 buyer doesn't have a specific subject year.
    - community_id / municipality_id → from the buyer's geoContext
                 (already in state).

  The matcher's output (SOLD listings filtered to a tax band) is
  exactly what the operator's CORRECT design demands. The current
  buyer derivation in lib/charlie/buyer-tax-match.ts builds the
  WRONG thing — it samples ACTIVE listings instead of querying
  Closed/SOLD ones.

  The seller matcher's geo-level cascade (community → muni → area)
  also produces a tier signal (`sourceTier`); buyer tiles could
  carry that same tier badge, which today they cannot because they
  have no `sourceTier`.
