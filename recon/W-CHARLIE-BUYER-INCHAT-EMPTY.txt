STEP 4 — VERDICT
==========================================================================

  ROOT CAUSE: field-shape mismatch in the in-chat render layer.
  ComparableCard reads camelCase only; the buyer comp data is
  snake_case (raw mls_listings columns from /api/geo-listings,
  passed through unchanged by get_comparables). Every field read
  returns undefined → tiles render with placeholder icon + '—'
  price + '—' address + no metadata + no click handler.

  Email + lead page render correctly because their renderers were
  authored with DUAL-SHAPE fallbacks (`c.camelCase || c.snake_case`)
  on every field — they were defensively written to handle both
  the seller-estimate camelCase shape AND the mls_listings snake_case
  shape. ComparableCard was authored only for the seller-estimate
  shape and pre-Chunk-2 was unreachable from the buyer path (gated
  off whenever a sellerEstimate block existed).

  EXACT FIX OPTIONS (Claude's call once authorized — do NOT pick now):

  Option A (cheapest, mirrors email/lead-page pattern):
    Edit ComparableCard to add `|| c.snake_case` fallbacks on every
    field read (lines 76-77, 80-83, 101-103, 131, 177, 180-184).
    Symmetric with the email + lead-page renderers. ~10 line edits,
    surgical. Risk: zero — seller-estimate camelCase shape still
    matches first, snake_case only fires when the camelCase field
    is missing.

  Option B (cleanest, single source of truth):
    Normalize the buyer comp shape at the push site (useCharlie.ts:
    610-612). Build a small mapper that converts snake_case →
    camelCase BEFORE pushing block.listings. Risk: low, but
    requires touching the push, the email POST body shape, and
    the lead-page plan_data write — three sites, three churns;
    invalidates the dual-shape reads in email/lead-page (which
    aren't broken).

  Option C (defer; only fix the count-display):
    Treat the empty-tile bug as a UI-styling issue and just hide
    the section when block.listings is empty. Doesn't address the
    actual data the buyer wanted to see. NOT recommended.

  Recommended: Option A. Surgical, mirrors the existing pattern in
  email/lead-page, makes ComparableCard's render layer the third
  dual-shape-aware consumer (matching the convergence principle).

  COUNT-DISPLAY FIX (separate concern):
    Gate the block render on `block.listings.length > 0` in
    ResultsPanel.tsx:411-419 so an empty comparables block is
    SKIPPED entirely (no "0 found" header). This is a one-line
    change and would suppress the cosmetic confusion if the
    2-block hypothesis (step 3) is correct.
