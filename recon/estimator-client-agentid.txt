W-ESTIMATOR-CLIENT-AGENTID — RECON (read-only)
Date: 2026-06-17
Hypothesis: the click doesn't reach the server because agentId is
empty on the rendered client page, firing the silent skip.

═══════════════════════════════════════════════════════════════════════════
HEADLINE
═══════════════════════════════════════════════════════════════════════════

  THE SALE OFFER AND LEASE OFFER BUTTONS ARE DEAD ON walliam.ca AT
  THE CLIENT RENDER GATE — BEFORE the server is ever called. The
  modal that holds the submit form is gated on `agent && ...` in the
  parent client component, and on walliam.ca (`isHero=true`) the
  parent server component passes `agent={null}`. Click sets
  `showOfferModal = true`, but `{showOfferModal && agent && (...)}`
  evaluates to falsy → the modal never mounts → no form ever
  renders → no submit ever fires → no x-tenant-id null-check ever
  reached → 0 leads + 0 activities.

  THE GET ESTIMATE BUTTON renders correctly. Its modal mount condition
  is `{showEstimatorModal && (...)}` with NO `agent &&` qualifier,
  and `agentId={agent?.id || walliamAgentId || ''}` resolves to
  King Shah's UUID via the walliamAgentId fallback. The
  HomeEstimatorResults `if (!agentId) skip` at L180 does NOT fire
  for walliam.ca. If the operator filled and submitted the estimator
  form, that path should reach the server.

  ROOT CAUSE (Sale + Lease Offer): mismatched client gate at
    app/property/[id]/HomePropertyPageClient.tsx:276
        {showOfferModal && agent && (
          <OfferInquiryModal ... agent={agent} />
        )}
  The `agent &&` short-circuits to null on the WALLiam hero path.
  Either the gate should also check walliamAgentId (or read agent
  from a hero-aware resolution helper), or OfferInquiryModal should
  receive a resolved `agentId` string (`agent?.id || walliamAgentId
  || ''`) instead of a full `agent` object — same pattern the
  HomeEstimatorBuyerModal mount at L268 already uses.

R3 — CONFIRM THE SKIP IS THE GATE (not the header)
═══════════════════════════════════════════════════════════════════════════

  Two distinct skips in play. Different layers, different effects:

  SKIP TYPE 1 — Modal-mount short-circuit (the actual bug)
    app/property/[id]/HomePropertyPageClient.tsx:276
        {showOfferModal && agent && (<OfferInquiryModal .../>)}
    On walliam: agent=null → the JSX expression becomes `null` →
    React renders NOTHING for that branch. NO inner component
    mounts. NO form. NO submit handler bound. NO network request.
    NO server action invocation.

  SKIP TYPE 2 — agentId early-return inside the submit handler
    (already-known, NOT the cause here)
    app/estimator/components/HomeEstimatorResults.tsx:180
        console.log('🔍 DEBUG EstimatorResults:', { agentId, buildingId, buildingName })
        if (!agentId) {
          console.log('⚠ No agentId - skipping lead creation (public context)')
          setSubmitted(true)
          setShowContactForm(false)
          return
        }
    Same pattern in EstimatorResults.tsx:111-116.
    This is INSIDE the form-submit handler. It fires AFTER the user
    clicks "Submit" inside an already-rendered form. The early-return
    prevents the server-action calls (submitActivityFromForm +
    submitLeadFromForm) from happening. UI sets submitted=true so
    the user sees a success state even though nothing was written.

    On walliam.ca, the Get Estimate path has agentId=King Shah's
    UUID → this skip does NOT fire. Estimator should work.

  WHICH SKIP IS HITTING THE OPERATOR:
    Operator's "0 activity, 0 lead in last 10 min" → no rows from
    EITHER server action means the server actions were not invoked
    at all. That rules out the L180 skip on the estimator (since
    agentId is truthy there) AND it rules out the x-tenant-id header
    null-check (since the server actions weren't even reached).

    The only remaining explanation among the candidates we've traced
    is SKIP TYPE 1 — the user clicked Sale Offer or Lease Offer,
    the parent gate at L276 evaluated `showOfferModal && agent` to
    `true && null` = falsy, the modal never rendered, and the
    operator saw "nothing happened" and reported failure.

    ALTERNATIVE for the Get Estimate path: user opened the modal
    (which IS visible because that gate has no `agent &&`) but
    didn't actually complete the form submit, or completed it but
    the inner result was the post-submit success-without-write
    state from the `if (!agentId) skip` … which doesn't apply on
    walliam (agentId IS truthy). So Get Estimate would write rows
    if the user submitted.

  ▶ R3 VERDICT: SKIP TYPE 1 (the JSX render-gate at
    HomePropertyPageClient.tsx:276) is the gate firing for the
    Offer buttons. It happens BEFORE the server action call and
    BEFORE any agentId check inside the modal. The server never
    runs. The x-tenant-id header fix is irrelevant on this path —
    it never gets exercised because the modal never mounts.

ROOT CAUSE (per button)
═══════════════════════════════════════════════════════════════════════════

  SALE OFFER + LEASE OFFER buttons:
    File: app/property/[id]/HomePropertyPageClient.tsx
    Line: 276
    Gate: `{showOfferModal && agent && (...)}`
    On walliam.ca (isHero=true), parent server component passes
    `agent={isHero ? null : agent}` at HomePropertyPage.tsx:308,
    so the client receives `agent=null`. The mount short-circuits;
    OfferInquiryModal never renders; no form, no submit, no server
    action, no row in either table.
    The mount gate must be walliam-aware — either also accept
    walliamAgentId (mirror the Estimator's L268 fallback chain)
    OR refactor OfferInquiryModal to take an `agentId: string`
    prop instead of `agent: AgentObject` so the same fallback
    pattern can be reused without keeping a full agent object.

  GET ESTIMATE button:
    File: app/property/[id]/HomePropertyPageClient.tsx
    Line: 263-273
    Gate: `{showEstimatorModal && (...)}`  (no agent qualifier)
    agentId={agent?.id || walliamAgentId || ''} at L268 resolves to
    King Shah's UUID on walliam. HomeEstimatorResults.tsx:180's
    `if (!agentId) skip` does NOT fire. The path is healthy
    end-to-end. If the operator submitted the form and still got 0
    rows, suspects shift to:
      - Did the user actually complete the multi-step modal flow
        (auth gate / VIP / actual contact form submit)?
      - If the user clicked outside the form mid-flow, no submit
        fires (no server-action POST in Network tab).
      - If the user IS authed and clicked submit and STILL got no
        row, then we're back to the x-tenant-id question and
        should look at server logs for
        '[submitLeadFromForm] x-tenant-id header missing'.
