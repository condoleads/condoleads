ROOT-CAUSE STATUS
═══════════════════════════════════════════════════════════════════════════

  ELIMINATED — proven NOT the cause:
    (a) `if (!agentId)` silent skip       — both surfaces resolve to a
                                            valid UUID.
    (b) FK reject on lead insert          — BEGIN/ROLLBACK insert
                                            succeeds end-to-end.
    (c) Missing Resend api_key            — present + verified.
    (d) WALLiam tenant disabled/inactive  — is_active=true,
                                            lifecycle_status=active.

  STILL OPEN — not exercised by this probe:
    (e) Middleware not setting x-tenant-id on the SUBMIT (server-
        action) request. The probe ran resolution against the DB
        directly; it bypassed the server-action layer that
        submitLeadFromForm uses. submitLeadFromForm reads
        headers().get('x-tenant-id') in app/actions/submitLeadFromForm.ts:51;
        if that header is missing on the form-submit, the action
        returns {success:false, error:'Tenant context unavailable.'}
        BEFORE reaching getOrCreateLead. The probe cannot reproduce
        this from a Node script — it requires a real browser submit
        with devtools open.
    (f) Network/fetch failure on the submit (CORS, 401, 500 on the
        server action). Same reason — needs live capture.
    (g) The buyer-email send (lib/actions/leads.ts:351+) is gated on
        `workingDoc?.hasAnyDocSection`. If the WorkingDoc the client
        passes lacks all 3 sections (comparableSold + taxMatch +
        competing all empty), buyer email is SILENTLY SKIPPED at
        L351. Could explain "no email" while still creating a lead.
    (h) The agent-email send is wrapped in try/catch at L329-337 with
        4 caught error classes (TenantEmailNotConfigured,
        TenantEmailFailed, generic). If ANY throws, the send is
        skipped silently and the lead row is still created. Could
        explain "no email but lead exists" (operator should check
        the leads table for the contact_email under WALLiam — if
        rows exist, S2 is the EMAIL leg, not the lead leg).

  HALF-OPEN observation worth flagging:
    OBSERVATION 1 — `send_from` column on tenants is NULL.
      The column exists. If lib/email/sendTenantEmail (not read in
      this probe) requires send_from to be non-null and falls back
      to throwing TenantEmailNotConfigured when null, then EMAIL
      sends silently fail for the WALLiam tenant. Worth a quick
      grep on sendTenantEmail to confirm whether send_from is
      consulted vs. computed from email_from_domain + brand_name.
