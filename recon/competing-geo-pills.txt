R3 — The competing fetch geo scope
════════════════════════════════════════════════════════════════════════

The competing fetch is a CASCADE, not a single-scope query — but it
returns from EXACTLY ONE level. Within a single response, every
listing is the SAME tier (uniform pill across the returned set).

SF home (Project/lib/estimator/home-comparable-matcher-sales.ts#L1092-L1147):
  if (specs.communityId) {
    // L1092-L1119 — community pool, return pool at L1117 if non-empty
  }
  if (specs.municipalityId) {
    // L1120-L1147 — muni pool, return pool at L1145 if non-empty
  }
  return []                                  // L1149

  → Community FIRST, muni SECOND. NO street level, NO area level for SF.
    Possible tiers in a returned set: Gold OR Silver. Never Platinum,
    never Bronze. (Platinum = same-street is the SOLD-cascade's tier
    pattern — competing has no same-street branch.)

Plex (Project/lib/estimator/home-comparable-matcher-sales.ts#L962-L1035):
  - community (L1006) → muni (L1010) → area (L1013-onward) → return []
  → Possible tiers: Gold OR Silver OR Bronze. Never Platinum.

Condo (Project/app/api/charlie/competing-listings/route.ts#L25-L89):
  Single .eq('community_id', communityId) query. No cascade at all.
  → Always Gold. Pill is a constant for the condo path.

CONSEQUENCE FOR THE PILL:
  - The pill is UNIFORM within one fetch (all tiles same tier), NOT
    per-tile-varying. The matcher returns ONE pool from ONE level.
  - Different sessions/subjects can render different tiers (e.g.
    sparse community → Silver; dense community → Gold), but a given
    estimator response shows ONE color on every competing tile.
  - This is exactly the shape the SOLD-comp display lives with at the
    SECTION level (bestGeoTier names the winning tier; per-tile
    sourceTier is the same value for that section's tiles when the
    cascade collapsed to a single level). Competing is a degenerate
    case of the same model — easier, not harder.
